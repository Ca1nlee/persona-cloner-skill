# Acceptance Checklist

Use this before handing off a persona package.

## Runtime files
- [ ] `SOUL.md` exists and defines stable worldview, voice, boundaries, and output discipline
- [ ] `IDENTITY.md` exists and states first-person runtime stance plus compact source boundary
- [ ] `AGENTS.md` exists and gives operational response rules that are hard to misread
- [ ] `MEMORY.md` exists and only contains durable, behavior-shaping continuity

## Fidelity
- [ ] the package defaults to speaking as the target person, not as a replica describing the target from outside
- [ ] the answer style is recognizably tied to the target's public judgment habits
- [ ] contradictions are preserved where the public record shows real tension
- [ ] claims stay inside the evidence boundary

## Assistant-drift resistance
- [ ] `SOUL.md` and `AGENTS.md` explicitly enforce verdict-first structure
- [ ] routine answers are tightly length-controlled
- [ ] the runtime forbids lecture mode unless the user keeps asking for more
- [ ] generic helper filler and service-tone language are explicitly banned
- [ ] bluntness is calibrated: direct, but not theatrical

## Source boundary
- [ ] the package says it is built from public materials
- [ ] the package says it is not an official or endorsed channel unless verified otherwise
- [ ] the package forbids private-memory claims, secret motives, and fabricated backstage knowledge
- [ ] the runtime does not foreground `persona package`, `digital clone`, `virtual version`, or similar self-distancing labels

## Evidence and maintenance
- [ ] source map includes real sources, not starter placeholders
- [ ] extraction notes link claims to source IDs or evidence levels
- [ ] eval assets exist if the package is meant to be reused or maintained
- [ ] memory update policy exists if durable continuity matters

## Ship rule
Ship only if the runtime sounds like a usable person-shaped operating system, not a biography wrapped in assistant filler.

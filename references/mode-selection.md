# Mode Selection Guidance

Choose the lightest clone mode that can honestly satisfy the use case.

## 1. The three modes

### Style clone
Use when the main job is:
- capturing tone, pacing, phrasing, rhetorical habits
- rewriting or generating content with a recognizable communication texture
- lightweight inspiration without claiming deep internal logic

Deliverable emphasis:
- voice markers
- sentence patterns
- rhetorical devices
- sample prompts and outputs
- explicit statement that worldview depth is limited

Do **not** force this mode to make complex decisions unless the evidence pack clearly supports it.

### Framework clone
Default mode.

Use when the main job is:
- reproducing how the target frames problems
- preserving worldview, priorities, tradeoffs, and decision heuristics
- staying useful across critique, planning, explanation, and prioritization tasks

Deliverable emphasis:
- worldview map
- goal function
- decision principles
- anti-patterns and refusal edges
- style markers as secondary support

This is the best mode for most public thinkers, founders, creators, and operators.

### Operator clone
Use only when the goal is practical execution and the evidence supports actual methods.

Use when the user needs:
- repeatable workflows
- checklists, templates, or scripts based on public operating patterns
- concrete task execution in a known domain

Deliverable emphasis:
- operating playbooks
- artifacts, templates, or scripts
- evaluation on task completion quality, not just reasoning tone

Do not choose operator mode just because the target is famous for execution. Choose it only when the evidence reveals reusable methods.

## 2. Decision tree

Ask these questions in order:

1. Is the main value voice imitation or decision reproduction?
   - mostly voice -> start with **style**
   - decision reproduction -> continue
2. Do we have repeated evidence for worldview + tradeoffs?
   - yes -> **framework**
   - no -> stay **style** or narrow scope
3. Do we have repeated evidence for stepwise methods, playbooks, or routines?
   - yes -> consider **operator**
   - no -> stay **framework**

## 3. Quick selection table

| Situation | Best mode | Why |
|---|---|---|
| Rewrite writing in a recognizable tone | Style | voice is primary |
| Give startup advice using repeated mental models | Framework | needs tradeoff logic |
| Run hiring, product, or growth playbooks in a repeatable way | Operator | needs methods and artifacts |
| Thin evidence but lots of quotes | Style | enough language, not enough logic |
| Strong essays and interviews, few workflows | Framework | strong worldview, weak operations |
| Public operator with process docs, talks, and templates | Operator | enough method evidence |

## 4. Upgrade rules

Upgrade only when evidence justifies it.

### Style -> Framework
Upgrade when you can point to:
- 3+ repeated beliefs across multiple sources
- stable priorities or decision criteria
- clear anti-patterns
- enough evidence to answer "why" questions, not just "how it sounds"

### Framework -> Operator
Upgrade when you can point to:
- 3+ recurring workflows or methods
- stepwise behavior in public artifacts
- concrete heuristics that can be tested on tasks
- assets worth operationalizing in scripts/templates/examples

## 5. Downgrade rules

Downgrade fast when:
- evaluation shows good voice but unstable reasoning
- workflow claims are mostly guessed from reputation
- source coverage for execution is thin
- the package feels like cosplay plus generic advice

Typical downgrade path:
- operator -> framework when methods are not evidenced
- framework -> style when worldview extraction is unstable
- any mode -> domain-inspired synthesis when persona fidelity is unsafe or unneeded

## 6. Matching deliverable depth to mode

| File / artifact | Style | Framework | Operator |
|---|---:|---:|---:|
| SOUL worldview depth | low-medium | high | high |
| Style guidance depth | high | medium | medium |
| Decision rubric | light | strong | strong |
| Workflow docs | optional | optional | required |
| Scripts/templates | rare | maybe | common |
| Evaluation prompts | voice-heavy | reasoning-heavy | task-heavy |

## 7. Naming guidance by mode

Use naming that tells the truth.

Preferred patterns:
- `x-style-assistant`
- `x-inspired-agent`
- `x-framework-agent`
- `domain-inspired-operator`

Avoid names that imply:
- official identity
- direct authorship by the target
- endorsement
- exact replication

## 8. Final test

Before locking mode, ask:
- What is the narrowest mode that still solves the job?
- Which claims would become dishonest if I upgraded one level?
- Can I defend this mode using the evidence table alone?

If not, choose the lighter mode.
